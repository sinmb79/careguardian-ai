## 1. WHY — 문제 정의

### 1.1 핵심 문제: 돌봄 지식의 단절

보호자(부모, 배우자, 자녀)가 수십 년간 체화한 돌봄 지식은 **그 사람의 머릿속에만 존재**한다.
보호자가 사망·입원·인지 저하로 사라지면, 돌봄 지식도 함께 사라진다.

- "파란 컵이 아니면 물을 안 마셔요"
- "목요일 오후에 불안해하면 산책이 필요해요"
- "이 소리를 내면 화가 난 게 아니라 배가 고픈 거예요"
- "약은 반드시 밥 먹은 직후에, 우유와 함께"

이 지식이 없는 후속 돌봄자는 **처음부터 시행착오를 반복**하며, 피돌봄자는 혼란·불안·위험에 노출된다.

### 1.2 수치로 본 현실

**발달장애인**
- 등록 발달장애인 약 26만 명, 일상생활 완전 자립 가능 비율 10.2%
- 68.8%가 부모 도움으로 생활, 주돌봄자의 58.7%가 부모
- 보호자 중 60대 이상 46%, 우울증 의심 52%
- 어떤 복지기관도 이용하지 못하는 성인 발달장애인 약 4.5만 명 추정
- 평생계획 필요성 인식 3.61/4점, 실제 준비 수준 극히 미흡
- 가장 큰 두려움: '부모 사후 돌봐줄 사람이 없을 것' (49.6%)
- 노후 경제적 준비 '전혀 안 됨' 92.6%

**고독사**
- 2024년 고독사 사망자 3,924명 (전년 대비 7.2% 증가)
- 40~64세 중장년 비중 최다
- 19세 이상 국민 3명 중 1명이 사회적 고립 상태

**치매**
- 65세 이상 치매 환자 약 100만 명 (2025 기준)
- 주돌봄자의 대부분이 가족 1인

### 1.3 왜 아직 이런 서비스가 없는가

1. **시장 논리 부재** — 대상자에게 구매력이 없다. VC 투자 대상이 아님.
2. **암묵지 인식 부재** — 돌봄자가 자신의 지식을 '기록 대상'으로 인식하지 못함.
3. **기술-복지 단절** — IT 업계는 도메인 지식이 없고, 복지 현장은 종이+전화.
4. **개인정보·윤리 장벽** — 건강·행동·선호 데이터 전달의 법적 프레임 불명확.
5. **정책이 이제야 시작** — 장애인개인예산제 2026년 시행, 사회적 고립 대응 국정과제화.

### 1.4 유사 서비스 부재 확인

| 서비스 | 하는 일 | 안 하는 일 |
|---|---|---|
| MARly Caregiver OS (US, 2026) | 가정 내 돌봄 조율, 건강 변화 감지 | 돌봄자 부재 시 지식 전달 |
| Caring Village (US) | 가족 간 캘린더·복약 공유 | 돌봄 매뉴얼 구조화 |
| 똑똑안부확인서비스 (KR) | IoT/AI 안부 확인 | 돌봄 지식과 무관 |
| 서울시 스마트플러그 (KR) | 전력 감지 → 안부 | 단방향 감시형 |
| 복지로/정부24 (KR) | 서비스 신청 포털 | 개인 돌봄 정보 관리 없음 |

**결론: "돌봄 지식의 구조화 → 보존 → 인수인계" 도구는 국내외 전무.**

## 2. WHAT — 프로젝트 정의

### 2.1 한 문장 정의

> 보호자가 살아있을 때 자신의 모든 돌봄 지식을 AI에게 학습시키고,
> 보호자가 떠난 뒤에도 그 AI가 피돌봄자 곁에 남아 일상을 안내하며,
> 후속 돌봄자에게 돌봄 지식을 인수인계하는 시스템.

### 2.2 대상 (범용 모듈 구조)

| 케이스 | 피돌봄자 | 돌봄 지식 보유자 | 위기 시나리오 |
|---|---|---|---|
| A | 성인 발달장애인 | 부모 (고령/말기질환) | 부모 사망·입원 |
| B | 치매 노인 | 배우자/자녀 1인 | 주돌봄자 사망·입원 |
| C | 독거 어르신 | 본인 (인지 저하 전) | 인지 악화 후 자기결정 불가 |
| D | 중증 희귀질환자 | 가족 1인 | 돌봄자 번아웃·이탈 |

공통 구조: **한 사람의 머릿속에만 있는 돌봄 지식 → 구조화 → 전달**

### 2.3 핵심 가치 3개

1. **디지털 유산 (Digital Care Legacy)** — 돌봄 지식을 구조화해서 영속시킨다.
2. **후견 AI (Guardian AI)** — 보호자를 대체하는 것이 아니라, 보호자의 지식으로 후속 돌봄을 돕는다.
3. **돌봄 연속성 (Care Continuity)** — 보호자가 바뀌어도 피돌봄자의 일상은 끊기지 않는다.

### 3.2 핵심 모듈 상세

#### Module 1: 돌봄 매뉴얼 빌더 (Care Manual Builder)

**입력 방식:**
- 음성 대화 (Gemma 4 + Whisper/Google STT)
- 텍스트 입력
- 사진 첨부 (약 사진, 좋아하는 물건 등)
- AI 능동 질문 ("식사 시간은 보통 몇 시인가요?", "잠들기 전 루틴이 있나요?")

#### Module 2: 일상 동반자 (Daily Companion)

피돌봄자가 매일 사용하는 AI 인터페이스.

**UI 원칙:**
- 극단적 단순화 — 큰 아이콘, 큰 글씨, 최소 터치
- 그림 기반 일과표 (시각 스케줄러)
- 음성 우선 인터페이스 ("안녕, 오늘 뭐 해?")
- 피돌봄자 인지 수준에 따라 UI 복잡도 조절 가능

**기능:**
- 시각 일과표 (아침 → 오후 → 저녁, 아이콘 기반)
- 복약 알림 + 확인 (사진 인증 → 돌봄자에게 전송)
- 원터치 SOS (위치 + 돌봄 체인 순서 발신)
- 간단한 돈 관리 ("오늘 쓸 수 있는 돈: 5,000원" 시각화)
- 감정 체크인 (이모지 선택 → 기록)
- AI 음성 말벗 (매뉴얼 기반 개인화된 대화)
- 위치 공유 (선택적, 후견인에게만)

**케이스별 변형:**

| 기능 | 발달장애 | 치매 노인 | 독거 어르신 |
|---|---|---|---|
| UI 복잡도 | 최소 (그림) | 최소 (큰 글씨) | 보통 |
| 음성 비중 | 높음 | 높음 | 선택 |
| 반복 안내 | 일과 중심 | 기억 보조 | 일정 중심 |
| SOS | 후견인 체인 | 보호자 체인 | 이웃+119 |
| 위치 | 배회 방지 | 배회 감지 | 선택적 |

#### Module 3: 돌봄 릴레이 (Care Relay)

돌봄 지식을 후속 돌봄자에게 안전하게 전달하는 시스템.

**릴레이 체인:**
```
주돌봄자(보호자)
  ↓ [트리거 발생]
1차: 지정 백업 돌봄자 (가족, 후견인)
  ↓ [1차 미응답/부재]
2차: 기관 (발달장애인지원센터, 지역복지관)
  ↓ [2차 미응답]
3차: 지자체 (읍면동 복지플래너)
  ↓ [긴급]
4차: 119 + 경찰
```

**트리거 유형:**
- 수동: 보호자가 직접 "인수인계 시작" 버튼
- 반자동: 보호자 입원 시 사전 설정된 자동 알림
- 자동: IoT 이상 감지 + 보호자 연락 두절 시
- 사후: 사망 신고 연동 (행정정보 공유 API, 향후)

**인수인계 시 전달되는 것:**
- 돌봄 매뉴얼 전체 (암호화 해제 → 지정 후견인에게)
- 최근 30일 생활 패턴 요약
- 현재 복약 스케줄
- 진행 중인 복지서비스 목록
- 긴급 연락처 체인

#### Module 4: 홈 허브 IoT (Home Guardian)

가정 내 생활 패턴을 학습하고 이상을 감지하는 IoT 시스템.

**하드웨어 (DIY, 타겟 비용 5~8만원):**
- ESP32-S3 (메인 컨트롤러) — ~8,000원
- Reed 스위치 x2 (현관문, 냉장고) — ~3,000원
- PIR 모션 센서 x2 (거실, 화장실) — ~4,000원
- 온습도 센서 (DHT22) — ~3,000원
- 비상 버튼 — ~2,000원
- 스마트플러그 (기성품, 선택) — ~15,000원
- 전원 어댑터 + 케이스 — ~5,000원

**Optional 상위 구성 (Raspberry Pi):**
- Raspberry Pi 5 4GB — Gemma 4 E2B 로컬 구동 가능
- 로컬에서 패턴 분석 + 자연어 이상 보고 생성
- 카메라 모듈 (선택, 낙상 감지용 — 프라이버시 주의)

**감지 로직:**
```
[센서 데이터 수집]
     ↓
[생활 패턴 베이스라인 학습] (최초 2주)
     ↓
[실시간 비교]
     ↓
[이상 판단]
  ├─ Level 1 (관찰): 평소보다 1시간 늦은 기상 → 로그만 기록
  ├─ Level 2 (확인): 냉장고 12시간 미개봉 → AI 음성 안부 확인
  ├─ Level 3 (알림): 24시간 활동 없음 → 돌봄 체인 1차에 알림
  └─ Level 4 (긴급): 비상 버튼 / 48시간 무활동 → 119 + 전체 체인
```

**핵심 원칙: 감시가 아닌 습관 학습**
- 카메라는 기본 비포함 (선택적 낙상 감지 모듈만)
- 센서는 "존재 여부"만 감지, "무엇을 하는지"는 감지하지 않음
- 모든 데이터 로컬 저장, 클라우드 전송 없음
- 피돌봄자 동의 필수 (후견인 대리 동의 가능)

#### Module 6: 비전 가디언 (Vision Guardian)

피돌봄자의 **눈**이 되어 세상을 읽어주는 멀티모달 AI 안내 시스템.
Gemma 4 E2B/E4B의 네이티브 비전+오디오 기능을 활용.

**6-A. 외출 모드 (Phone Camera + Earphone)**

```
[피돌봄자가 이어폰 착용 + 폰 카메라 켜기]
     ↓
[Gemma 4 E4B: 카메라 프레임 실시간 분석]
     ↓
[이어폰으로 음성 안내]
  ├─ 길 안내: "횡단보도예요. 신호 빨간불이니까 기다려"
  ├─ 간판 읽기: "왼쪽에 GS25 편의점이 있어"
  ├─ 위험 감지: "앞에 계단 있어, 조심해"
  ├─ 매뉴얼 연동: "여기 자주 오던 빵집이야. 엄마 기록에 크림빵 좋아한다고 돼있어"
  └─ 길 잃음 감지: "평소 경로에서 벗어났어. 돌봄자에게 알릴까?"
```

핵심: 단순한 카메라 앱이 아니라, **돌봄 매뉴얼과 연결된** 안내.
"이 사람이 좋아하는 가게", "이 사람이 피해야 하는 장소" 등 개인화 정보 반영.

**기술 구현:**
- Gemma 4 E4B 비전: 카메라 프레임 → 장면 설명 (on-device, 오프라인)
- OCR: 간판, 표지판, 버스번호 읽기 (다국어 네이티브)
- 물체 감지: 계단, 차량, 횡단보도, 장애물 바운딩 박스
- TTS: 장면 설명 → 이어폰 음성 출력 (한국어)
- GPS + 매뉴얼 연동: 자주 가는 장소 학습, 경로 이탈 감지

**6-B. 실내 모드 (Home Camera + Speaker)**

```
[가정 내 카메라 (RPi + USB 카메라 / IP 카메라)]
     ↓
[Gemma 4 E2B on RPi: 영상 분석]
     ↓
[스피커로 음성 안내]
  ├─ 일과 안내: "수호야, 12시야. 점심 먹을 시간이야"
  ├─ 안전 감지: "가스레인지 켜진 지 30분 됐어. 확인해봐"
  ├─ 복약 안내: "빨간 약통에서 큰 알약 하나 먹을 시간이야"
  ├─ 낙상 감지: "움직임이 없어. 괜찮아? 대답 없으면 이모한테 연락할게"
  └─ 루틴 유도: "밥 먹었으니 이 닦으러 가자"
```

핵심: **카메라가 감시하는 게 아니라, 부모처럼 말을 걸어주는 것.**
"밥 먹었어?"가 아니라 "밥 먹었으니 이 닦으러 가자."

**기술 구현:**
- Raspberry Pi 5 + USB Camera or IP Camera (기존 CCTV 재활용 가능)
- Gemma 4 E2B (LiteRT-LM on RPi): 프레임 분석
- 행동 인식: 식사 중, 취침, 앉아있음, 서있음, 쓰러짐
- 사물 인식: 약통, 가스레인지, 물컵 등 매뉴얼에 등록된 물건
- 스피커 출력: Bluetooth 스피커 or RPi 직결 스피커
- 프라이버시: 영상 저장 안 함 (실시간 분석 후 프레임 폐기), 로컬 전용

**6-C. 미래 확장: AI 안경 (AI Glasses)**

```
[AI 안경 (카메라 + 마이크 + 골전도 스피커)]
     ↓ BLE
[Phone (Gemma 4 E4B)]
     ↓
[실시간 비전+음성 안내]
```

- AI 안경 자체는 카메라+마이크+스피커만 탑재 (연산은 폰 위임)
- 이미 시중에 $50~200대 오픈소스 AI 안경 프레임 존재 (ESP32-S3 + OV2640)
- 폰에서 Gemma 4 E4B가 영상 처리 → 골전도 스피커로 안내
- Phase 3+ 이후 확장 항목

**프라이버시 특별 조항 (비전 모듈):**

| 원칙 | 구현 |
|---|---|
| 영상 비저장 | 실시간 분석 후 프레임 즉시 폐기. 녹화 없음. |
| 로컬 전용 | 모든 비전 처리는 on-device. 클라우드 전송 없음. |
| 타인 얼굴 비식별 | 얼굴 감지 시 피돌봄자 외 인물은 블러/무시 |
| 동의 기반 | 실내 카메라는 피돌봄자(또는 후견인) 동의 필수 |
| 물리적 제어 | 카메라 ON/OFF 물리 스위치 + LED 표시등 |

#### Module 5: 복지 매칭 엔진 (Welfare Matcher)

피돌봄자의 상황에 맞는 복지서비스를 자동으로 찾아주는 시스템.

**입력:**
- 매뉴얼의 장애 유형, 등급, 소득 수준, 가구 형태, 지역
- 현재 이용 중인 서비스 목록

**출력:**
- 추가 신청 가능한 서비스 목록 + 신청 방법 안내
- 긴급 상황 시 이용 가능한 긴급돌봄 서비스

**연동 대상:**
- 복지로 API (복지서비스 모의계산)
- 정부24 (서비스 신청)
- 장애인개인예산제 바우처 시스템 (2026~)
- 지자체 고독사 예방 시스템
- 장기요양등급 관련 서비스

## 5. PRIVACY & SECURITY

### 5.1 설계 원칙

1. **Local-First**: 모든 돌봄 데이터는 기본적으로 디바이스에만 저장
2. **Zero-Knowledge Cloud**: 클라우드 백업 시 E2E 암호화, 서버는 평문 접근 불가
3. **Consent-Based Relay**: 인수인계는 보호자의 사전 지정 + 피돌봄자 동의(후견인 대리 가능)
4. **Minimal IoT Data**: 센서는 "존재 여부"만 감지, 영상/음성 녹화 없음 (기본)
5. **Right to Delete**: 피돌봄자 또는 후견인이 언제든 전체 데이터 삭제 가능

### 5.2 돌봄 매뉴얼 접근 권한

| 역할 | 매뉴얼 읽기 | 매뉴얼 쓰기 | IoT 데이터 | 릴레이 설정 |
|---|---|---|---|---|
| 주보호자 | ✅ | ✅ | ✅ | ✅ |
| 백업 돌봄자 | ✅ (지정 후) | ❌ | ✅ (요약만) | ❌ |
| 후견인 | ✅ (릴레이 후) | ✅ (인수인계 후) | ✅ | ✅ |
| 활동지원사 | ✅ (관련 섹션만) | ❌ | ❌ | ❌ |
| 시설 담당자 | ✅ (릴레이 후) | ❌ | ❌ | ❌ |
| 피돌봄자 | 제한적 | ❌ | ❌ | ❌ |

### 5.3 법적 고려사항

- 개인정보보호법: 민감정보(건강, 장애) 처리 동의 필수
- 성년후견제도: 피후견인의 데이터 처리에 후견인 권한 인정
- 장애인권익옹호: 돌봄 매뉴얼이 피돌봄자의 자기결정권을 존중하도록 설계
- 고독사예방법: 이상 감지 및 긴급 알림의 법적 근거

## 6. UX SCENARIOS

### Scenario A: 발달장애 아들을 둔 말기암 어머니

```
Phase 1 — 매뉴얼 구축 (어머니 생존 시)
  어머니: "수호는 아침에 일어나면 항상 초록색 수건으로 세수해야 해요"
  AI: "아침 루틴 - 세면 섹션에 기록했습니다. 세수 후 다음 순서는 뭔가요?"
  어머니: "양치 끝나면 아침밥인데, 밥은 꼭 노란 그릇에..."
  → 2~3개월에 걸쳐 매뉴얼 완성

Phase 2 — 릴레이 설정
  어머니: 후견인(이모)을 백업 돌봄자로 지정
  어머니: 시설(A 그룹홈) 담당자를 2차 릴레이로 지정
  → 릴레이 트리거: 입원 시 자동 알림

Phase 3 — 어머니 입원
  시스템: 어머니 폰 48시간 비활성 + 병원 GPS 감지
  → 이모에게 카카오톡 알림: "수호 어머니 연락 두절. 확인 필요"
  → 이모 확인 후 인수인계 시작
  → 이모에게 돌봄 매뉴얼 전체 열람 권한 부여

Phase 4 — 어머니 사후
  이모: "수호가 목요일마다 불안해하는데 어떻게 해야 해?"
  AI: "매뉴얼에 따르면 수호는 목요일 오후에 산책을 하면 안정됩니다.
       어머니께서 '공원 벤치에서 비둘기 보는 것을 좋아한다'고 기록하셨습니다."
  
  수호의 폰: (매일 아침)
  AI: "수호야, 좋은 아침! 오늘 목요일이야. 오후에 공원 가자. 😊"
```

- [ ] 복약 알림
- [ ] 로컬 암호화 저장
- **산출물:** Android PWA, 보호자 1명 + 피돌봄자 1명 시나리오 동작

**목표:** 돌봄 릴레이 + 후견인 모드

- [ ] 후견인 대시보드

- [ ] Raspberry Pi + USB 카메라 실내 비전 모듈

- [ ] 매뉴얼 연동 장소 안내 + 경로 이탈 감지
- [ ] IoT + 비전 데이터 → 돌봄 매뉴얼 자동 보강

### Phase 3: Welfare & Community
**목표:** 복지 매칭 + 돌봄자 커뮤니티 + AI 안경 확장
- [ ] 복지로/정부24 API 연동
